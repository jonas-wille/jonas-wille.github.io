A Python program to turn any file into an MP4 and back, by storing bytes as colour blocks.
It was created with the intention of seeing whether it would be possible to store data persistently on a platform like YouTube. Unfortunately, in its current implementation the process is far too inefficient and slow for any meaningful practical use. The silly idea for it came to me after watching this rather excellent YouTube video.

# The Idea

The original plan was that the program reads the raw bytes of a file and to each possible byte it then associates a colour. However, since there are 256 bytes, we would need 256 different colours. Finding 256 colours a computer can accurately distinguish, was not something I wanted to subject myself to, instead I decided to just go with nibbles (4 bits) so I would only be dealing with 16 colours. From the colour information of the nibbles the program then draws images with colours blocks representing the nibble information. These images are then used as frames to generate a video. The resulting video looks something like this:  

<img src="https://user-images.githubusercontent.com/84239673/173181642-a41df83f-6584-44d7-898a-600cb1c6ac98.gif" width="40%" height="40%"/>

We can decode a video generated by the program by reversing the process. Split the video into individual images and extract the nibble information from the blocks in the image and then string it all back together in a file.

# Limitations
The process of reading a file and generating a video from it / reversing it is rather slow. Files with a few 100KB will already take 10-20 seconds depending on the machine. Therefore, encoding larger files is not recommended. This is something that could maybe be alleviated with parallelisation, but the video encoding process will always be quite a large bottleneck.
\
\
The 16 colours are distinct enough that in combination with the generous block size the format is resilient to compression like the one done on YouTube. This is of course though not a guarantee and there are surely scenarios where information would be lost.
